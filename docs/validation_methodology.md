# RENA-3 Emulation Validation Methodology

## 1. Overview

### Why Validation Is Necessary

This project trains a depth-of-interaction (DOI) regression model on waveform data acquired with an Arizona CAEN DT5730 digitizer but deploys it on the Stanford CZT PET system, which uses the **RENA-3 ASIC** for signal readout. The two readout chains process the same physical CZT signals through fundamentally different hardware paths:

| Stage | Arizona CAEN (training data source) | Stanford RENA-3 (deployment) |
|---|---|---|
| Shaping | None (raw waveform) | RC-CR analog filter |
| Triggering | Software threshold on raw samples | Leading-edge discriminator on shaped signal |
| Peak capture | Software maximum over full waveform | Sample-and-hold after trigger |
| Digitization | 14-bit, 62.5 MHz | 12-bit, internal clock |

If the emulation is wrong, the feature values computed from CAEN data will be **systematically offset** from what RENA-3 hardware would produce for identical physical events. A model trained on mismatched features will fail in deployment even if training metrics look good.

The validation suite here confirms that each stage of `Rena3Emulator` produces outputs consistent with published RENA-3 specifications and known CZT detector physics.

## 3. Validation Components

Six validation plots are generated by `run_scientific_validation()` in `pipeline.py` and `scripts/run_validation.py`. Each is described below with its scientific justification and pass criterion.

---

### V1 — RC-CR Kernel Shape

**Output file**: `results/validation/emulation/rccr_kernel.png`

**What**: Plots the RC-CR impulse response kernel `h(t)` computed by `Rena3Emulator._build_rccr_kernel()` vs time, overlaid with the theoretical curve.

**Why**: The kernel governs the entire shaped waveform. If `τ` is wrong or the normalization is off, all downstream peak amplitudes and therefore all 15 features are systematically incorrect.

**Theoretical form**:

```
h(t) = (t / τ²) · exp(−t / τ),   τ = T_peak / 2
```

The peak of `h(t)` occurs at `t = τ = T_peak / 2`... wait — differentiating and setting to zero: `dh/dt = (1/τ² − t/τ³) exp(−t/τ) = 0` → `t = τ`. So the kernel peaks at `t = τ = T_peak / 2`. However, the **convolution** of the kernel with a step function (the CSP output) peaks at `t = T_peak = 2τ`. The kernel itself peaks at `τ`.

**How**: The kernel array is plotted directly. The peak sample index is compared to the expected location `τ / DT_S`.

**Expected result**: Single-lobed peak with maximum at `t ≈ τ = 1.4 µs`; decays to near-zero by `t = 10τ = 14 µs`.

**Pass criterion**:
```python
|argmax(kernel) * DT_S - tau_s| < DT_S * 2
```

**Reference**: RENA-3 datasheet §3.2; Gu dissertation §2.3.

---

### V2 — Waveform Transformation

**Output file**: `results/validation/emulation/waveform_transform.png`

**What**: Overlays raw (baseline-subtracted) and RC-CR shaped waveforms for 4 representative events on the primary anode channel.

**Why**: The CZT charge-sensitive preamplifier (CSP) produces **step-function** signals: the waveform rises rapidly when charge arrives and then holds at a plateau. A simple `argmax` on the raw waveform would find the plateau edge, not the true peak. RC-CR shaping differentiates the step into a single peaked pulse, enabling reliable peak amplitude extraction. If shaping is not working, the raw step and shaped signal will look identical.

**How**: `Rena3Emulator.debug_process_chunk()` returns `raw_f64`, `shaped`, and `kernel` without triggering ADC scale fitting. The first 4 events are plotted with dual y-axes (raw in blue, shaped in orange).

**Expected result**: Raw traces show a plateau after initial rise; shaped traces show a single peaked pulse that returns toward zero. The shapes should be clearly distinct.

**Pass criterion**: For the shaped signal on the primary anode channel, the peak-to-mean ratio must exceed 2.0:
```python
shaped_peak / abs(shaped_mean) > 2.0
```

**Reference**: Gu dissertation §2.3; standard RC-CR shaping theory.

---

### V3 — Energy Spectrum

**Output file**: `results/validation/data_quality/energy_spectrum.png`

**What**: Histogram of `E_anode_primary` (primary anode ADC count) across all events in the validation subset, plotted on a log-scale y-axis.

**Why**: The dataset uses a Cs-137 source (662 keV γ-rays). A correctly functioning peak extraction pipeline should produce a visible photopeak cluster in the energy spectrum. If peak extraction is broken — e.g., extracting baseline noise instead of true peaks — the spectrum will be a structureless low-count distribution with no photopeak.

**How**: Histogram with 200 bins over the full ADC range (0–4095). The y-axis is logarithmic to reveal both the photopeak and Compton continuum simultaneously.

**Expected result**: A cluster or peak at some consistent ADC value corresponding to 662 keV full-energy deposits, above a declining Compton continuum.

**Pass criterion**: Visual inspection. Automatic thresholding is not applied because the exact photopeak position depends on per-strip gain variation (documented range: 1.06×–4.83×) and trigger threshold settings. The absence of any discernible structure would indicate a broken pipeline.

**Reference**: Cs-137 decay scheme (662 keV primary γ); Gu dissertation §4.

---

### V4 — CAR vs DOI Scatter

**Output file**: `results/validation/physics/car_vs_doi.png`

**What**: Scatter plot of `CAR = E_cathode_primary / E_anode_primary` (feature index 2 in `FEATURE_NAMES`) vs ground-truth DOI in mm.

**Why**: CAR is the **established primary DOI indicator** for single-sided CZT strip detectors (He et al. 1997; Gu et al. 2014). The physical mechanism: cathode signals arise from electron motion (high µτ), while anode signals include both electron and hole contributions. As interaction depth increases (moving from anode toward cathode), more hole trapping occurs on the anode side, reducing E_anode relative to E_cathode, so CAR increases monotonically with depth. A non-monotonic or flat CAR-DOI relationship means either the emulation is computing incorrect peak amplitudes or the ground-truth labelling (step_index × 8 mm) is wrong.

**How**: Scatter plot, alpha=0.03, rasterized. Spearman rank correlation ρ(CAR, DOI) is computed and printed.

**Expected result**: Clear positive trend — points should drift upward as DOI increases from anode (0 mm) toward cathode (320 mm).

**Pass criterion**:
```python
scipy.stats.spearmanr(X_raw[:, car_idx], y).statistic > 0.3
```

**Reference**: He et al. *NIM A* 388 (1997) 180–185; Gu et al. *IEEE NSS* 2014; Gu dissertation §3.

---

### V5 — CAR Box Plot per Step

**Output file**: `results/validation/physics/car_boxplot.png`

**What**: Box-and-whisker plot of CAR grouped by motor step index (i.e., by DOI ground-truth depth).

**Why**: The scatter plot (V4) can be visually ambiguous due to high event density. The box plot summarizes the CAR distribution at each discrete depth step, making the calibration curve directly visible. It also reveals whether the variance is stable across depths (expected) or increases/decreases (would indicate gain drift or threshold effects).

**How**: Events are grouped by step index using the ground-truth DOI label divided by `MOTOR_STEP_MM`. One box per step, ordered left-to-right by increasing depth.

**Expected result**: Median CAR increases left-to-right across all sampled steps.

**Pass criterion**: Median CAR at the deepest step > median CAR at the shallowest step:
```python
median_CAR[deepest_step] > median_CAR[shallowest_step]
```

**Reference**: Same as V4.

---

### V6 — Charge Trapping Signature

**Output file**: `results/validation/physics/charge_trapping.png`

**What**: Scatter plot of `E_anode_primary` vs ground-truth DOI, showing the decrease in anode signal with increasing depth.

**Why**: CZT has strongly asymmetric carrier transport. The electron µτ product is ~10⁻² cm²/V, but the **hole µτ is ~5×10⁻⁵ cm²/V** — roughly 200× worse. For interactions near the cathode, holes must travel the full detector thickness before being collected by the anode. With such low µτ, a significant fraction of holes are trapped in transit, reducing the total induced charge on the anode. This is the fundamental physical mechanism that makes CZT single-carrier detectors and that creates the DOI-dependent energy variation. If the anode amplitude is **flat** or **increases** with depth, it means peak extraction is capturing noise rather than true signal amplitudes.

**How**: Scatter plot, alpha=0.03, rasterized. Spearman rank correlation ρ(E_anode_primary, DOI) is computed and printed.

**Expected result**: Visible negative slope — higher DOI correlates with lower anode amplitude.

**Pass criterion**:
```python
scipy.stats.spearmanr(X_raw[:, ea_idx], y).statistic < -0.1
```

**Reference**: He et al. *NIM A* 388 (1997) 180–185; Knoll, *Radiation Detection and Measurement*, 4th ed., §13.8.

## 4. Running the Validation

### Prerequisites

```bash
pip install -e ".[dev]"
```

Data must be present at the path specified in `configs/default.yaml` under `ingest.data_dir`.

### Command

```bash
python scripts/run_validation.py --config configs/default.yaml --steps 0 10 20 30 40
```

The `--steps` argument selects which motor-position indices (0–40) to include. Using 5 steps spanning the full depth range (0 mm to 320 mm) is sufficient to verify monotonic CAR behaviour while keeping runtime short.

### Expected Output Files

```
results/validation/
├── emulation/
│   ├── rccr_kernel.png           # V1
│   └── waveform_transform.png    # V2
├── physics/
│   ├── car_vs_doi.png            # V4
│   ├── car_boxplot.png           # V5
│   └── charge_trapping.png       # V6
├── data_quality/
│   └── energy_spectrum.png       # V3
└── reports/
    └── validation_report.json    # all computed metrics
```

### Interpreting `validation_report.json`

```json
{
  "kernel_peak_error_s": 1.6e-08,
  "kernel_pass": true,
  "waveform_peak_to_mean": 8.3,
  "waveform_pass": true,
  "spearman_car_doi": 0.61,
  "car_doi_pass": true,
  "car_median_monotone": true,
  "car_boxplot_pass": true,
  "spearman_eap_doi": -0.34,
  "charge_trapping_pass": true
}
```

All boolean fields should be `true` for a passing validation. The Spearman ρ values are the primary quantitative indicators of physical correctness.

## 5. Limitations and Known Caveats

### ADC Scale Auto-Fitting

The RENA-3 hardware uses a fixed gain setting per run. Our emulation auto-fits the ADC scale from the 99th percentile of the **first processed chunk**. This means:
- The scale is data-dependent, not a true hardware constant
- If the first chunk happens to contain unusually large or small events, the scale will be off for all subsequent chunks
- This is a known approximation; a future improvement would be to calibrate scale against the photopeak position in the spectrum

### Trigger Threshold

The 10% leading-edge threshold (`trigger_threshold_fraction = 0.10`) is a software approximation of the RENA-3 hardware discriminator level. The actual discriminator level in the Stanford system depends on per-channel register settings not recorded in the dataset. The 10% value is tuned to avoid noise triggering while maintaining sensitivity to real pulses.

### Gain Variation

Per-strip gain variation of 1.06×–4.83× is documented in the CZT detector characterization data. Phase 1 does **not** apply gain correction — all channels are treated as having identical gain. This will broaden feature distributions and may weaken the CAR-DOI correlation compared to a gain-corrected analysis. Gain correction is planned for Phase 2.

### Scan Geometry Requirement

V4, V5, and V6 require the motor to scan along the **crystal depth axis** — that is, the beam must enter through the crystal edge with a 0.5 mm slit defining the interaction depth layer, and the motor must move perpendicular to the cathode face across the ~10 mm crystal thickness.

A **lateral scan** (motor parallel to anode strips) produces ρ ≈ 0 for all three DOI physics checks. This is physically correct behaviour, not a bug: CAR is a depth indicator and carries no information about lateral position. All three checks will fail together whenever `y = step_index × 8 mm` represents lateral position rather than depth.

Signs that a dataset is a lateral scan:
- Total motor range >> 10 mm (e.g., 41 steps × 8 mm = 320 mm spans ~40 crystal modules laterally)
- Filename contains `Parallel_to_anode` or similar
- V4/V5/V6 all fail simultaneously while V1/V2 pass

Expected depth-scan setup: beam perpendicular to cathode face, motor range ≈ 0–10 mm per crystal, filename would contain `Perpendicular_to_anode` or similar. When V4/V5/V6 all fail, the validation script prints a diagnostic note and records a `scan_geometry_note` field in `validation_report.json`.

The ADC scale fix (per-event max 99th percentile) remains valuable for any scan type; it improves feature quality regardless of the correlation outcome.

### Waveform Model Mismatch

The CAEN digitizer acquires at 62.5 MHz for 2001 samples (32 µs window). The RENA-3 in the Stanford system may use different timing parameters. Any difference in pre-trigger window length or total acquisition window affects baseline subtraction and therefore all downstream amplitudes.
